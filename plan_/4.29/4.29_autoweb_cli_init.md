# 4.29 AutoWeb 接入 dp_cli 初始化执行计划

生成日期：2026-04-29

## 0. 结论先行

目标不是一次性删除 AutoWeb 的 Python Coder，而是把“LLM 写 DrissionPage Python 并由 `exec()` 执行”逐步替换为“LLM 产出结构化 dp_cli action，由受控适配层执行”。推荐路线：

1. 先在 AutoWeb 内新增 `DPCLIExecutor` 适配层，不改现有主流程。
2. 再加 `execution_mode == "dp_cli"` 双轨分支，让 Coder 可以输出 action JSON。
3. 稳定后让 Observer 优先使用 dp_cli `snapshot --mode agent_summary`。
4. 爬取类任务优先迁移到 `extract` / `batch-detail-extract`。
5. 最后再做 ActionCache，旧 CodeCache 保留到命中率下降且 dp_cli 覆盖面足够。

这个顺序可以每一步单独验收，失败时也能回退到当前 AutoWeb 流程。

## 1. 当前事实

### AutoWeb 当前链路

- 图入口在 `core/graph_v2.py`，流程是 `Observer -> Planner -> CacheLookup -> Coder/Executor -> Verifier`。
- `core/nodes.py` 中：
  - `cache_lookup_node()` 当前只检索 Python `generated_code`，命中后直接跳 `Executor`。
  - `coder_node()` 当前使用 `ACTION_CODE_GEN_PROMPT` 生成 Python 代码，写入 `generated_code` 和 `_code_source = "llm"`。
  - `executor_node()` 当前调用 `BrowserActor.execute_python_strategy()`，后者在 `skills/actor.py` 中使用 `exec(strategy_code, {}, local_scope)`。
  - `verifier_node()` 根据 `execution_log` 和当前 URL 判断是否继续、回 Observer，或进入 RAGNode 存缓存。
- 状态字段在 `core/state_v2.py`，目前只有 `generated_code`、`execution_log`、`coder_retry_count`、`_code_source`、`_cache_hit_id` 等代码缓存相关字段。
- HITL 已经在 `core/graph_v2.py` 的 `interrupt_before=["Executor"]` 上中断，这对 action 审核很有利。

### dp_cli 当前能力

远端仓库：https://github.com/1122414/drissionpage-cli

本地仓库：`E:\GitHub\Repositories\drissionpage-cli`

从 README 和本地代码看，dp_cli 的主契约是：


统一返回：

```json
{
  "ok": true,
  "session": "demo",
  "action": "snapshot",
  "data": {},

```

本地版本已经有这些命令：

- `find`

注意：远端 GitHub 页面能看到的是基础契约；本地检出的代码能力更完整。AutoWeb 接入时应做版本/命令探测，不要默认所有环境都有 `batch-detail-extract`。

## 2. 目标架构

```text
AutoWeb
  Observer
    -> 阶段 1 仍用旧 DOM skeleton
    -> 阶段 3 优先读 dp_cli snapshot agent_summary

  Planner
    -> 决定下一步任务
    -> 标记本步适合 dp_cli 还是旧 Python Coder

  CacheLookup
    -> 阶段 1 保持 CodeCache
    -> 阶段 5 增加 ActionCache

  Coder
    -> python_code 模式：保留旧 generated_code
    -> dp_cli 模式：输出 generated_action JSON

  Executor
    -> python_code 模式：BrowserActor.exec 旧逻辑
    -> dp_cli 模式：DPCLIExecutor 执行 action

  Verifier
    -> 统一看 execution_log / dpcli_result
    -> dp_cli 错误码可确定性路由
```

## 3. 状态协议

在 `core/state_v2.py` 给 `AgentState` 增加：

```python
generated_action: Optional[Dict[str, Any]]
execution_mode: Optional[str]       # "python_code" | "dp_cli"
dpcli_session: Optional[str]
dpcli_result: Optional[Dict[str, Any]]
dpcli_snapshot: Optional[Dict[str, Any]]
_action_source: Optional[str]       # "llm" | "action_cache" | None
_action_cache_hit_id: Optional[str]
_failed_action_cache_ids: List[str]
```

兼容策略：

- 老流程默认 `execution_mode = "python_code"`。
- `_code_source` 保留给旧 CodeCache，不要马上复用成 action source。
- `generated_code` 和 `generated_action` 不要同时作为有效执行对象；Coder 每次更新时要清掉另一个。

建议 action JSON：

```json
{
  "skill": "click",
  "params": {
    "ref": "e12"
  },
  "reason": "点击 snapshot 中的搜索按钮"
}
```

允许的第一批 skill：

```text
open, snapshot, find, click, type, expand, list-items, extract, resolve-locator, session.inspect
```

第二批再开放：

```text
batch-detail-extract, eval
```

其中 `eval` 必须默认关闭，只作为受控 fallback。

## 4. 里程碑 M1：只接执行器，不动主流程

目标：AutoWeb 可以稳定调用 dp_cli，但 Coder/Executor 主流程不变。

### 改动

新增 `skills/dpcli_executor.py`：

```python
class DPCLIExecutor:
    def __init__(self, session="autoweb", headless=False, python_executable=None, cwd=None): ...
    def open(self, url, wait_time=None): ...
    def snapshot(self, mode="agent_summary", ref=None, depth=None, wait_time=None): ...
    def find(self, text=None, locator=None, wait_time=None): ...
    def click(self, ref=None, locator=None, wait_time=None): ...
    def type_text(self, ref=None, locator=None, text="", wait_time=None): ...
    def expand(self, ref, depth=2): ...
    def list_items(self, group_ref, sample_size=5): ...
    def extract(self, target_ref, schema=None, limit=None): ...
    def resolve_locator(self, ref): ...
    def session_inspect(self): ...
    def batch_detail_extract(...): ...
```

实现要求：

- 使用 `subprocess.Popen` 或 `subprocess.run` 调 `python -m dp_cli`。
- `cwd` 默认指向 `E:\GitHub\Repositories\drissionpage-cli`，后续用配置项覆盖。
- `python_executable` 优先读配置，默认可用当前 Python；在本机建议为 `D:\Anaconda\envs\dp-cli\python.exe`。
- stdout 必须按 UTF-8 解 JSON。
- 非 0 exit code 也要尽量解析 stdout 中的 JSON，因为 dp_cli 会返回结构化 error。
- 返回值统一为 dict，不向上抛普通 CLI 错误。
- 给超时返回：

```json
{
  "ok": false,
  "action": "click",
  "error": {"code": "timeout", "message": "...", "details": {"timeout": 60}}
}
```

配置项加到 `config.py`：

```python
DPCLI_ENABLED = _env_bool("DPCLI_ENABLED", False)
DPCLI_CWD = os.getenv("DPCLI_CWD", r"E:\GitHub\Repositories\drissionpage-cli")
DPCLI_PYTHON = os.getenv("DPCLI_PYTHON", "python")
DPCLI_SESSION = os.getenv("DPCLI_SESSION", "autoweb")
DPCLI_HEADLESS = _env_bool("DPCLI_HEADLESS", False)
DPCLI_TIMEOUT_SECONDS = int(os.getenv("DPCLI_TIMEOUT_SECONDS", "60"))
DPCLI_BATCH_TIMEOUT_SECONDS = int(os.getenv("DPCLI_BATCH_TIMEOUT_SECONDS", "900"))
```

新增测试：

  - mock subprocess：验证命令拼接、JSON 解析、错误解析、超时分支。
- `scripts/smoke_dpcli_executor.py`
  - 真实跑 `open about:blank` 或本地 fixture。

验收命令：

```bash

```

M1 完成标准：

- 不修改 `coder_node()`、`executor_node()` 主逻辑。
- 旧 AutoWeb 流程完全不受影响。
- `DPCLIExecutor.snapshot()` 能返回 `schema_version`、`page_identity`、`index`。

## 5. 里程碑 M2：Executor 加 dp_cli action 分支

目标：不改 Coder 之前，先证明 Executor 能执行手写/测试注入的 action。

### 改动

在 `core/nodes.py` 增加：

```python
def _execute_dpcli_action(state: AgentState, config: RunnableConfig) -> dict:
    ...
```

在 `executor_node()` 开头加分支：

```python
execution_mode = state.get("execution_mode") or "python_code"
if execution_mode == "dp_cli":
    return _executor_dpcli_branch(state, config)
```

`_executor_dpcli_branch()` 负责：

1. 读取 `generated_action`。
2. 校验 action schema。
3. 调用 `DPCLIExecutor`。
4. 把结果写入：
   - `dpcli_result`
   - `execution_log`
   - `current_url`
   - `verification_result` 失败时先给确定性结果
5. 成功 goto `Verifier`。
6. 失败按错误码路由：
   - `ref_stale` / `ref_not_found` -> goto `Observer`，提示重新 snapshot。
   - `invalid_ref_type` / `invalid_input` -> goto `Coder`，让 LLM 修 action。
   - `element_not_found` / `element_not_interactable` -> goto `Observer`。
   - `browser_config_error` / `timeout` / `unexpected_error` -> goto `ErrorHandler`。

错误结果写入示例：

```python
verification_result = _build_verification_result(
    is_success=False,
    is_done=False,
    summary="dp_cli action failed: ref_stale",
    source="executor",
    failure_scope="local",
    failed_action=json.dumps(generated_action, ensure_ascii=False),
    failed_locator=generated_action.get("params", {}).get("ref", ""),
    evidence=json.dumps(dpcli_result.get("error"), ensure_ascii=False),
    fix_hint="重新 snapshot 后使用新的 ref"
)
```

需要同步改 `main.py`：

- HITL 审核 Executor 前，若 `execution_mode == "dp_cli"`，展示 `generated_action`，不要只展示 `generated_code`。
- 手动编辑时：
  - 旧模式继续编辑 `temp_code_edit.py`。
  - dp_cli 模式编辑 `temp_action_edit.json`。
- `_build_manual_verification_result()` 增加 action 风险扫描，至少拦截：
  - `eval`
  - `open` 非 http/https/about/file 白名单
  - `batch-detail-extract` item 数过大且无 limit

新增测试：

- `test/test_dpcli_executor_node.py`
  - mock `DPCLIExecutor.click` 成功 -> goto `Verifier`。
  - mock `ref_stale` -> goto `Observer`。
  - mock `invalid_input` -> goto `Coder`。

验收命令：

```bash
python -m unittest test.test_dpcli_executor_node -v
python -m py_compile core/nodes.py core/state_v2.py main.py skills/dpcli_executor.py
```

M2 完成标准：

- 可以通过 `app.update_state(..., {"execution_mode": "dp_cli", "generated_action": ...})` 手动注入 action 并执行。
- 旧 `generated_code` 路径不变。

## 6. 里程碑 M3：Coder 输出 dp_cli action JSON

目标：让 LLM 不再写 Python，而是为适合的步骤输出 action。

### 改动

新增 `prompts/dpcli_action_prompts.py`：

```python
DPCLI_ACTION_GEN_PROMPT = """
你只能输出 JSON 对象，不要 Markdown。
可用 skill:
- open: params.url
- snapshot: params.mode/ref/depth
- find: params.text 或 params.locator
- click: params.ref 或 params.locator
- type: params.ref 或 params.locator, params.text
- expand: params.ref, params.depth
- list-items: params.group_ref, params.sample_size
- extract: params.target_ref, params.schema, params.limit
- resolve-locator: params.ref

优先使用 snapshot 中的 ref。
e* 用于 click/type，r* 用于 expand/list-items/extract。
一次只输出一个 action。
"""
```

修改 `coder_node()`：

- 不要直接覆盖旧逻辑。
- 根据 `state.execution_mode` 或配置 `DPCLI_ENABLED` 分支：

```python
if _should_use_dpcli_action(state):
    return _dpcli_coder_node(...)
else:
    return _python_coder_node(...)
```

`_should_use_dpcli_action()` 初期规则：

- `DPCLI_ENABLED == True`
- 当前 plan 包含以下类型时优先 dp_cli：
  - 打开网址
  - 点击
  - 输入
  - 查找
  - 展开区域
  - 列表页提取
- 以下情况继续旧 Python：
  - 用户明确要求下载文件、复杂 API 调用、文件处理。
  - 需要调用 `toolbox.http_request()`。
  - dp_cli 未支持的浏览器动作。

Coder 成功输出时写：

```python
{
  "generated_action": action,
  "generated_code": None,
  "execution_mode": "dp_cli",
  "_action_source": "llm",
  "_code_source": None
}
```

JSON 解析失败：

- 第一次 retry 可让 Coder 再生成。
- 连续失败 2 次则 fallback 到旧 Python Coder。

新增测试：

- `test/test_dpcli_action_prompt.py`
  - 模拟 LLM 返回 fenced JSON，能解析。
  - 模拟非法 JSON，能进入 retry/fallback。
  - 确保 `generated_code` 被清空。

验收命令：

```bash
python -m unittest test.test_dpcli_action_prompt -v
python -m unittest test.test_dpcli_executor_node -v
```

M3 完成标准：

- 一个“打开 URL -> snapshot -> find -> type -> click”的任务可不生成 Python code。
- HITL 能看到 action JSON。

## 7. 里程碑 M4：Observer 读取 dp_cli snapshot

目标：把 AutoWeb 的 DOM skeleton/locator 策略逐步换成 dp_cli 的 `agent_summary`。

### 改动

在 `core/state_v2.py` 的环境状态增加：

```python
dpcli_snapshot_view: Optional[Dict[str, Any]]
```

在 `observer_node()` 里加可控分支：

```python
if DPCLI_ENABLED and DPCLI_OBSERVER_ENABLED:
    result = DPCLIExecutor(...).snapshot(mode="agent_summary")
    if result["ok"]:
        return Command(update={
            "dpcli_snapshot": result,
            "dpcli_snapshot_view": _compact_dpcli_snapshot(result),
            "current_url": result["data"]["page"]["url"],
            "dom_skeleton": _render_dpcli_snapshot_text(result),
            "_observer_source": "dp_cli",
        }, goto="Planner")
```

配置项：

```python
DPCLI_OBSERVER_ENABLED = _env_bool("DPCLI_OBSERVER_ENABLED", False)
DPCLI_OBSERVER_FALLBACK_TO_DOM = _env_bool("DPCLI_OBSERVER_FALLBACK_TO_DOM", True)
```

`_compact_dpcli_snapshot()` 只保留：

- `page.url`
- `page.title`
- `page_identity`
- `index.interactable_elements[:30]`
- `index.data_regions[:5]`
- `index.surface_index[:40]`
- `index.stats`
- 最近一次 `dpcli_result`

不要把 full snapshot 直接塞给 LLM。

Fallback：

- dp_cli snapshot 失败，且 `DPCLI_OBSERVER_FALLBACK_TO_DOM=True` -> 走旧 Observer。
- `browser_config_error` -> 标记 `verification_result`，去 `ErrorHandler`。

新增测试：

- `test/test_dpcli_observer_projection.py`
  - 给一个假 snapshot，确认压缩视图字段和长度。
  - snapshot 失败时确认 fallback 行为。

验收命令：

```bash
python -m unittest test.test_dpcli_observer_projection -v
```

M4 完成标准：

- Planner/Coder 可以基于 `interactable_elements` 和 `data_regions` 生成 action。
- Token 量低于旧 DOM skeleton 的常规输出。

## 8. 里程碑 M5：爬取链路优先 dp_cli 化

目标：把“列表页提取”和“详情页批量提取”从 Python 代码中拆出来，优先使用 dp_cli 原生命令。

### 推荐流程

```text
open(url)
snapshot(mode="agent_summary")
extract(target_ref=data_regions[0].ref)
find/click 下一页
snapshot(mode="agent_summary")
extract(target_ref=data_regions[0].ref)
batch-detail-extract(items=list_items)
save output
Verifier
```

### 改动

在 `DPCLIExecutor` 增加 `execute_action()`：

```python
def execute_action(self, action: dict) -> dict:
    skill = action["skill"]
    params = action.get("params") or {}
    ...
```

在 action schema 里开放：

```json
{
  "skill": "extract",
  "params": {
    "target_ref": "r2",
    "schema": ["title", "url"],
    "limit": 20
  }
}
```

`batch-detail-extract` 不建议直接由 Coder 随便发起，先由规则触发：

- 用户目标包含“详情”“点进去”“简介”“每一条详情”等。
- 最近 `extract` 结果中有 `detail_url` 或 `url`。
- `list_pages_extracted >= target_pages` 或用户只要求当前页。
- item 数量大于 0。

新增 `skills/dpcli_crawl_policy.py`：


输出文件建议统一写到 AutoWeb `output/`：

```text

logs/dpcli_progress_<session>.jsonl
```

注意：如果 dp_cli 命令在 `DPCLI_CWD` 下运行，传给 `--output-file` 的路径必须是绝对路径，避免写到 drissionpage-cli 仓库。

新增测试：

- `test/test_dpcli_crawl_policy.py`
  - 有 detail_url -> 生成 batch action。

验收命令：


- “爬取列表前 N 条”走 `extract`，不再生成 Python 循环。
- “点进每条详情”走 `batch-detail-extract`，不再让 LLM 对每个详情页单独推理。

## 9. 里程碑 M6：ActionCache 双轨缓存

目标：新成功 action 写入 ActionCache，旧 CodeCache 暂时保留。

### 改动

新增 `skills/action_cache.py`，结构参考 `skills/code_cache.py`，但字段改为：

```text
cache_id
user_task
goal
url_pattern
domain_key
snapshot_signature
task_type
action_json
result_summary
created_at
success_count
failure_count
```

检索 query：

```text
user_task + plan + current_url + compact snapshot data_regions/interactable summary
```

在 `cache_lookup_node()` 中先做：

```text
if DPCLI_ENABLED and ACTION_CACHE_ENABLED:
    try ActionCache
    if hit:
        generated_action = hit.action_json
        execution_mode = "dp_cli"
        _action_source = "action_cache"
        goto Executor
else:
    old CodeCache
```

失败标记：

- action cache hit 失败时写 `_failed_action_cache_ids`。
- 不影响 `_failed_code_cache_ids`。

存储点：

- `verifier_node()` 成功后，如果 `execution_mode == "dp_cli"` 且 `_action_source != "action_cache"`，进入 `RAGNode` 或直接后台保存 ActionCache。
- 旧 CodeCache 的保存条件只对 `execution_mode == "python_code"` 生效。

验收命令：

```bash
python -m unittest test.test_action_cache -v
python -m unittest test.test_verification_contract -v
```

M6 完成标准：

- 成功 action 可复用。
- 失败 action cache hit 会被短期屏蔽。
- 旧 CodeCache 不受影响。

## 10. dp_cli 侧建议补齐项

这些不阻塞 M1-M4，但会影响替换覆盖率：

1. 增加 `wait` 命令，替代 Coder 写 `time.sleep()`。
2. 增加 `scroll` 命令，支持 `to_bottom`、`to_top`、`by`。
3. 增加 `select` 命令，覆盖 `<select>` 和 combobox。
4. 给 `open/click/type` 的返回补齐 `page_identity`，便于 Verifier 做确定性判断。
5. 给 `batch-detail-extract` 增加更明确的 partial output contract，超时也能稳定返回已采集数据路径。
6. 给 `dp_cli --version` 或 `session inspect` 增加 `commands`/`capabilities`，AutoWeb 可以启动时探测能力。

## 11. 风险和回退

### 浏览器生命周期冲突

AutoWeb 现在通过 `BrowserDriver` 和 DrissionPage browser 对象管理浏览器；dp_cli 也会管理 session/profile/port。dp_cli 模式下必须明确：

- dp_cli action 只使用 dp_cli session 浏览器。
- 旧 Python Coder 只使用 AutoWeb browser。
- 双轨期间不要尝试让两边共享同一个 tab 对象。

### ref 失效

dp_cli 已有 `ref_stale`，AutoWeb 不要让 Coder 猜旧 ref。处理方式：

```text
ref_stale -> Observer 重新 snapshot -> Coder 生成新 action
```

### 安全边界

- LLM 不允许输出 subprocess/Python。
- `eval` 默认禁用。
- `batch-detail-extract` 必须有 item 数限制、timeout 和输出路径。
- HITL 审核 action 比审核 Python code 更简单，但仍要显示目标 URL、skill、params、风险提示。

### 版本差异

M1 中必须提供 `capabilities()`：

```python
{
  "open": true,
  "snapshot": true,
  "extract": true,
  "batch-detail-extract": false
}
```

探测方式：

- 首选 `python -m dp_cli --help` 解析 subcommands。
- 后续 dp_cli 增加 `--version --json` 后切换为结构化能力探测。

## 15. 判断是否可以删除旧 Python Coder 的条件

满足以下条件前，不删除旧 Python Coder：

- dp_cli 能覆盖至少 80% 常见交互：open/click/type/find/extract/scroll/wait/select。
- 列表页和详情页爬取有稳定输出路径。
- ActionCache 命中和失败黑名单已跑通。
- 至少 5 个历史场景通过：
  - 搜索类
  - 登录/注册类
  - 列表爬取类
  - 列表 + 详情页爬取类
  - 翻页类
- HITL 能完整审 action，而不是只审 code。
- 出错时能明确区分 action 选错、ref 失效、dp_cli 能力不足、浏览器环境问题。

结论：现在最稳的路线是双轨迁移。先把 dp_cli 当成 AutoWeb 的受控执行后端，等 action 模式和 snapshot 模式稳定后，再把旧 Coder 从默认路径降级成 fallback。
